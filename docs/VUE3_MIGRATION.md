# Стратегия миграции Vue 2 → Vue 3

Контекст: проект на Vue CLI 4, Vue 2.6, Vue Router 3, Vuex 3, Options API.

## Цели

- Сохранить работоспособность приложения на каждом этапе.
- Не смешивать миграцию с массовым рефакторингом бизнес-логики.

## Рекомендуемая последовательность

1. **Стабилизация текущей ветки** — зелёные `lint`, `test:unit`, `build`, исправленные критичные баги (роутинг, API-контракты, Vuex async).
2. **Переход на Vue 2.7** (опционально) — последние патчи Vue 2, совместимость с частью Vue 3 API при необходимости.
3. **Инвентаризация зависимостей** — список пакетов без поддержки Vue 3; замены (например `@vue/compat` для постепенной миграции).
4. **Сборка** — миграция с Vue CLI + Webpack 4 на Vite или Vue CLI 5 + Webpack 5 по отдельному эпику; обновление Node до LTS, зафиксированного в CI.
5. **Vue Router 4 + Vuex 4** (или Pinia) — по одному модулю: сначала bootstrap приложения, затем store, затем маршруты с `createRouter`.
6. **Компоненты** — порциями: замена фильтров, lifecycle hooks, `$listeners`, глобальной регистрации при необходимости.

## Риски

- Рост объёма при смешении миграции с крупным редизайном.
- Сторонние плагины без Vue 3-ветки.

## Критерий готовности к старту миграции

- CI зелёный, есть минимальные unit-тесты на критичные сценарии.
- Документирован целевой стек (Vite vs Vue CLI 5, Pinia vs Vuex 4).
